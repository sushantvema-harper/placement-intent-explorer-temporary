Strategic Objective

Build the data infrastructure that enables Jade — Harper's AI shopping agent — to be the sole autonomous agent that reliably carries a submission from dispatch through to a validated quote: failing gracefully, learning continuously, and working hand-in-hand with human operators to deliver higher throughput, faster time-to-quote, and increased revenue.

This is not about replacing operators. It's about giving them superpowers. Jade handles carrier follow-up, document routing, information gathering, and state tracking — so operators can focus on judgment calls, relationship building, and deal closing.

Strategic Context
1
Harper is an AI-native commercial insurance brokerage
Businesses come to Harper for commercial insurance. Harper doesn't underwrite — it shops. It manages the back-and-forth with carrier underwriters until quotes come back, then pitches those quotes to customers. Three operators currently process 200–300 emails per day across Gmail, Email Bot, BigBrother, and Whisper Flow.
2
The old system (BigBrother) had no mid-funnel visibility
BigBrother tracked status at the account level — flat, denormalized. It had "Pending Submissions" and "Quoted" but nothing in between. No way to see what's actively being shopped, which coverage is quoted vs. pending, or which carriers have gone silent. Urgency was tracked in operator memory.
3
The new system (CLS + MegaBro) tracks at coverage granularity with intent machines
CLS (Coverage Lifecycle System) decomposes each submission into CoverageIntents, SubmissionIntents, PlacementIntents, and RFQ Line Items — each with its own state machine. "GL quoted, PL still pending" is now a first-class state, not an implicit gap. MegaBro is the operator UI that surfaces this granularity.
4
PlacementIntent is the keystone entity
PlacementIntent bridges the gap between "we sent it" and "we got an answer." It owns the shopping phase per coverage per carrier group. Without it, CLS has no durable state for the Submitted → Quoted loop — the exact loop where Harper's operational time is consumed.
5
Jade (harper-claw) drives placement state transitions autonomously
Jade is a long-lived Temporal workflow per account. It classifies inbound emails (carrier and customer), checks Engram for existing data before asking customers anything, batches outbound communications, and drives RFQ Line Item transitions between info_requested ↔ under_review. When it can't handle something, it creates a structured HTA for the operator.
6
MiniOrq in Agora orchestrates the business logic
MiniOrq is the Temporal workflow in Agora that consumes Relay events and auto-transitions PlacementIntent state based on CASHMONEY quote events, email classifications, and operator actions. It is the bridge between the event bus and the CLS state machines.

The Phoenix Principle

PI is immutable once it reaches a terminal state. Recovery always creates new entities — a new SubmissionIntent, new RFQ, new RFQLI, new PI. The old PI becomes part of the audit trail: it records what was tried, with whom, and why it ended. This is the "phoenix" pattern — the old entity dies, a new one rises from it, carrying a reference back to the original.

Exception: When a PI is still in an active (non-terminal) state, updating in place is correct. The clearest example: a revised quote from the same carrier doesn't reset the PI — it adds a new Quote entity while the PI stays at quote_received. The PI tracks the carrier relationship; Quote tracks individual documents.

Remediation vs Phoenix Patterns

CLS uses two distinct loop patterns. PI v2 at RFQLI level needs both.

Key Design Change: disqualified_quote is NOT terminal in v2

disqualified_quote uses the remediation pattern — the same PI loops back to under_review for same-carrier quote revision. No new entity is created for same-carrier retries. Phoenix (new PI) only applies if the operator switches to a different carrier, in which case the PI transitions to remarketing (terminal) and a new PI is created downstream.

Consider adding revision_count (like IntakeWorkflow.ReopenedCount) to track how many times a PI has looped through the disqualified_quote → under_review remediation cycle.

RFQ Line Item State Machine

Tracks per-coverage outcome within an RFQ. When a line item reaches "quoted", a Quote entity is created. When ALL line items for an OLI reach "declined", remarketing becomes eligible. Currently transitioned by MiniOrc (Atlas Lackey) via POST /rfq-items/{id}/transition. Design target: derive status from PlacementIntent.

PriceIndication (not a separate entity)
Price indication lives as a status on RFQLineItem with two payload fields:
PriceIndicationPremium *float64 — carrier's soft ballpark premium
PriceIndicationLimits map[string]any — free-form limits (JSONB)
These fields are only populated when status = price_indication.
Quote.quote_type progression
When a Quote entity is created, its quote_type reflects the formality level:
price_indication → indication_verified → full_quote
A soft indication letter gets price_indication. A confirmed formal quote gets full_quote. indication_verified is the middle ground — indication confirmed by UW but not yet a formal document.

Entity Lifecycle: Create vs Update Decision Matrix

Full business logic for each phoenix scenario — what triggers entity creation, the exact data model implications, and unresolved design questions. Color coding matches PI v2 tab: green = create new, yellow = update in place, blue = mixed.

▲ CREATE NEW 1. Remarketing — All PIs declined, new carriers targeted
What triggers it

Operator clicks "Remarket" in MegaBro after all PIs for an OLI reach declined. CLS backtracks the CoverageIntent from placement → submission.

Entity creation chain

New SubmissionIntent (type=remarketing, prior_si_id=<original SI>) → Agora generates new packet for different carriers → new RFQ per new carrier → new RFQLI per coverage → new PI per RFQLI.

Old entity fate

Old PIs freeze at remarketing terminal state. Old RFQs: design question — soft-withdrawn or left in terminal state? Most likely left as-is since the carrier declined; withdrawal implies we rescinded an offer (we didn't).

Open question

Should old RFQs be explicitly transitioned to withdrawn? Or does the SI-level terminal state implicitly retire them? Recommendation: leave as declined (terminal), no explicit withdrawal needed.

▲ CREATE NEW 2. Rework — Defective packet (wrong docs / wrong entity)
What triggers it

Operator determines the submission packet was incorrect — wrong documents, wrong named entity, missing ACORD forms. NOT a carrier decision; it's an internal quality failure. CLS backtracks CI: placement → submission.

Entity creation chain

New SubmissionIntent (prior_si_id links back). Agora regenerates packet. New RFQ to same carrier → new RFQLI → new PI. Same underwriter context, fresh paper trail.

Old entity fate

Old PI freezes at rework terminal. Old RFQ remains — it represents the defective attempt and is part of the audit trail. Not withdrawn: we're not rescinding, we're acknowledging an error.

Key distinction from Remarketing

Remarketing = carrier said no. Rework = WE messed up. Different operator workflows, different SI types, different root-cause tracking.

▶ REMEDIATION LOOP 3. Quote Rework — disqualified_quote → under_review (same carrier, same PI)
What triggers it

A quote passed receipt but failed validation (disqualified_quote). Operator wants to contact the same carrier to get corrected quote data — not remarketing (same carrier, same coverage, no SI backtrack needed).

Remediation loop (not phoenix)

The PI loops back to under_review — same entity, same row, no new PI created. This follows the remediation pattern (like IntakeWorkflow: reviewing → remediation → collecting). Consider tracking revision_count to record how many times the loop has executed.

Why remediation, not phoenix

The submission was correct. The carrier relationship is intact. The carrier sent bad quote data (missing field, wrong premium format, wrong limits). We go back to the same UW — no new packet needed, no new entity. The PI is a parked holding state, not a dead end.

When phoenix still applies

If the operator decides to switch to a different carrier, the PI transitions to remarketing (terminal) and a new PI is created downstream. Remarketing = phoenix. Same-carrier quote revision = remediation loop.

▲ CREATE NEW 4. Unsolicited Coverage — Carrier bundles in Cyber when only GL requested
CASHMONEY's role

CASHMONEY's RFQMatcher detects a coverage line in the quote that has no matching RFQLI. It creates a synthetic RFQLI (is_synthetic=true) to house the unsolicited coverage. A PI is created only if a matching OLI can be resolved.

Operator workflow

Unsolicited PI enters a "review required" hold. Operator sees a work item: "Unsolicited Cyber quote from [Carrier] — accept or dismiss." If accepted, PI proceeds. If dismissed, PI transitions to withdrawn. Original GL PI is unaffected throughout.

▶ UPDATE IN PLACE 5. Revised Quote — Same carrier, revised premium ($12k → $9.5k)
Why update in place

The PI has NOT reached a terminal state. It's still in quote_received. The carrier relationship is unchanged — same UW, same carrier, same coverage, same RFQ. Only the quote document changed. PI tracks the placement relationship, not the document.

What changes

New Quote entity created (linked to same RFQLI). Old Quote entity preserved as history. PI stays at quote_received. CASHMONEY re-runs validation on the new Quote, may re-transition PI to disqualified_quote or approved_quote.

The core distinction

PI = carrier relationship + placement attempt. Quote = a specific document from that carrier. A carrier can send multiple quotes as part of one placement attempt. The PI only resets if the attempt itself fails terminally.

MegaBro display

Quote history panel shows all versions with timestamps. Most recent quote premium shown prominently. Operator can compare v1 vs v2 side-by-side.

~ MIXED 6. Split Submission — Carrier accepts GL but declines WC/PL
Normal flow (no phoenix)

This is the expected behavior of per-RFQLI PIs. GL RFQLI has its own PI (→ quote_received). WC/PL RFQLI has its own PI (→ declined). Independent state machines, no special handling needed.

When phoenix kicks in

If the operator then wants to remarket WC/PL to other carriers, that triggers the remarketing flow: new SI → new RFQ (different carrier) → new RFQLI (WC/PL) → new PI. The old WC/PL PI stays frozen at declined.

▲ CREATE NEW 7. Synthetic RFQ — Quote arrives with no matching RFQ (CASHMONEY creates synthetic)
Why this happens

Carrier emails a quote that was solicited informally (phone call, referral, existing relationship) and no formal RFQ was created in the system. CASHMONEY can't find a matching RFQ by carrier + OLI lookup, so it creates a synthetic RFQ to house the quote.

Entity creation

RFQ.is_synthetic=true → RFQLI → PI (if OLI resolved). PI immediately enters submitted but is flagged for operator review before advancing. No SubmissionIntent — the submission happened outside the system.

OLI resolution failure path

If CASHMONEY cannot resolve an OLI (no matching opportunity in CLS), the quote enters an "orphan queue" — a work item for operator to manually link to the correct opportunity. Once linked, normal PI flow resumes.

Synthetic vs unsolicited coverage

Synthetic = no RFQ existed at all. Unsolicited coverage = RFQ existed for one coverage (GL) but carrier quoted an additional one (Cyber). Different triggers, same is_synthetic=true flag on the generated RFQLI.

▲ CREATE NEW 8. Instant Quote Portal Retry — Portal declined (appetite), operator fixes data + resubmits
Why phoenix applies here

Portal responses are instant and deterministic — unlike a human UW email conversation, there's no ongoing negotiation. A portal decline is a system-generated final answer for that set of application inputs. Changing inputs = new submission attempt.

Entity creation

Old PI freezes at declined (with decline_reason=appetite). New RFQ to same portal (channel=instant_quote) with updated application snapshot → new RFQLI → new PI. Application data delta is captured in the new RFQ's metadata.

Portal-specific considerations

Portal channels skip acknowledged, info_requested, and price_indication states. The new PI starts at submitted and immediately transitions to either quote_received or declined — no waiting period.

Audit trail

MegaBro shows retry history with prior decline reason and what application fields changed. Helps operator understand why the appetite threshold was crossed and what correction was made.

MiniOrq Decision Tree: New Entity vs In-Place Update

Is the PI in a terminal state? (declined, rework, remarketing, withdrawn)
YES → Create new entities. Old PI is history. New SI (if backtrack needed) or new PI (if lean phoenix). Always link via prior_si_id / prior_pi_id.
Is the PI in an active state? (submitted, acknowledged, under_review, quote_received, ...)
YES → Is this a new carrier interaction (new quote document from same carrier)?
YES → Add new Quote entity. PI stays in place. This is the revised-quote scenario.
NO — it's a new carrier / new coverage → Create new RFQLI + PI. Original PI unaffected.
Is this an unsolicited quote (no prior RFQ)?
YES → CASHMONEY creates is_synthetic=true RFQ → RFQLI → PI (pending operator review).
